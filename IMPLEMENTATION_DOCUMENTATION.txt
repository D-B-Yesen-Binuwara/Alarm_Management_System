5. DEVICE FAILURE SIMULATION (IsSimulatedDown)
================================================================================

WHAT
----
A controlled mechanism to simulate device failures and recoveries for testing
and demonstrations. A boolean flag on the Device entity tells the heartbeat
scheduler to stop recording heartbeats for that device, causing the failure
detection service to naturally detect a timeout.

WHY
---
Random packet-loss simulation (10% miss rate) creates unpredictable failures
that are hard to demonstrate or test against specific scenarios. The IsSimulatedDown
flag gives full control over which device fails and when, while keeping the
monitoring logic unchanged — the API never directly sets a device to DOWN.

STAGES OF IMPLEMENTATION
-------------------------
Stage 1 — Entity & Migration:
  Added IsSimulatedDown (bool, default false) to Device.cs.
  Created SQL migration: Backend/Migrations/AddIsSimulatedDownToDevice.sql
    ALTER TABLE Device ADD IsSimulatedDown BIT NOT NULL DEFAULT 0;

Stage 2 — Scheduler Integration:
  HeartbeatSchedulerService was updated to skip heartbeat recording for any
  device where IsSimulatedDown = true (see Section 6).

Stage 3 — API Endpoints:
  Two new endpoints added to DeviceController:

  POST /api/devices/{id}/simulate-failure
    Sets device.IsSimulatedDown = true via UpdateAsync.
    The scheduler then stops recording heartbeats for this device.
    After 90 seconds, HeartbeatFailureDetectionService detects the timeout
    and sets Status = DOWN automatically.

  POST /api/devices/{id}/recover
    Sets device.IsSimulatedDown = false via UpdateAsync.
    The scheduler resumes recording heartbeats.
    HeartbeatFailureDetectionService detects the resumed heartbeat and
    sets Status = UP automatically.

FILES MODIFIED
--------------
- INMS.Domain/Entities/Device.cs                                    [MODIFIED]
- INMS.API/BackgroundServices/HeartbeatSchedulerService.cs          [MODIFIED]
- INMS.API/Controllers/DeviceController.cs                          [MODIFIED]
- Backend/Migrations/AddIsSimulatedDownToDevice.sql                 [NEW]

KEY PRINCIPLE
-------------
The API never directly changes device.Status. Status is always determined by
the monitoring logic (HeartbeatFailureDetectionService). The simulation flag
only controls whether heartbeats are recorded.

COMPLETE FAILURE WORKFLOW
--------------------------
POST /api/devices/5/simulate-failure
  → IsSimulatedDown = true
  → Scheduler skips device (no heartbeat recorded)
  → After 90s: FailureDetectionService detects timeout
  → device.Status = DOWN
  → SimulationEvent logged: HEARTBEAT_TIMEOUT

POST /api/devices/5/recover
  → IsSimulatedDown = false
  → Scheduler resumes recording heartbeats
  → FailureDetectionService detects valid heartbeat
  → device.Status = UP
  → SimulationEvent logged: HEARTBEAT_RECOVERED

6. HEARTBEATSCHEDULERSERVICE — BACKGROUND SERVICE
================================================================================

WHAT
----
An ASP.NET Core BackgroundService that runs every 30 seconds and records a
heartbeat for every active device in the system.

WHY
---
Real telecom monitoring systems poll devices periodically. This service simulates
that polling cycle. Without it, the failure detection service would have no
heartbeat data to evaluate.

FILE
----
- INMS.API/BackgroundServices/HeartbeatSchedulerService.cs [NEW]
- INMS.API/Program.cs [MODIFIED — AddHostedService registration]

EVOLUTION OF IMPLEMENTATION
-----------------------------
Version 1 (Initial — incorrect):
  Recorded a heartbeat for every device with Status == UP on every cycle.
  Problem: Heartbeats never stopped, so failure detection could never trigger.
  The scheduler itself was preventing failures from being detected.

Version 2 (Conceptual correction — random packet loss):
  Added 90% success rate simulation: random.Next(0, 100) > 10.
  Only recorded heartbeat if the device "responded" (90% chance).
  This allowed natural, random failures to occur.
  Problem: Failures were unpredictable and hard to control for demos/testing.
  Also noted: creating new Random() inside the method could produce predictable
  patterns; recommendation was to use a static field or Random.Shared.

Version 3 (Final — IsSimulatedDown controlled simulation):
  Removed random packet loss. Instead:
  - If device.IsSimulatedDown == true → skip (no heartbeat recorded)
  - Otherwise → record heartbeat regardless of current Status
  Recording heartbeat regardless of current Status (not just when Status == UP)
  is critical: it enables automatic recovery detection. If a device was DOWN
  and its IsSimulatedDown is cleared, the scheduler records a heartbeat, and
  the failure detection service sees a valid recent heartbeat and marks it UP.

FINAL LOGIC SUMMARY
--------------------
  foreach (var device in devices)
  {
      if (device.IsSimulatedDown) continue;  // skip — no heartbeat
      await heartbeatService.RecordHeartbeatAsync(device.DeviceId, "UP");
  }

CONFIGURATION
-------------
  Heartbeat interval: 30 seconds
  Runs continuously from application start

7. HEARTBEATFAILUREDETECTIONSERVICE — BACKGROUND SERVICE
================================================================================

WHAT
----
An ASP.NET Core BackgroundService that runs every 30 seconds (starting 45 seconds
after application start) and evaluates every device's heartbeat recency and
topology position to determine its correct status.

WHY
---
The scheduler records heartbeats but does not make status decisions. This service
is the "brain" of the monitoring system — it reads heartbeat timestamps, applies
the 90-second timeout rule, and propagates failures through the network topology.

FILE
----
- INMS.API/BackgroundServices/HeartbeatFailureDetectionService.cs [NEW, then REWRITTEN]
- INMS.API/Program.cs [MODIFIED — AddHostedService registration]

EVOLUTION OF IMPLEMENTATION
-----------------------------
Version 1 (Initial):
  Simple per-device heartbeat check. For each device, fetched the latest
  heartbeat and compared to current time. If > 90 seconds, marked DOWN.
  If device was DOWN and heartbeat resumed, marked UP.
  Problem: N+1 database queries (one heartbeat query per device).
  Problem: No topology awareness — all failures treated as direct device failures.
  Problem: HEARTBEAT_RECOVERED logged even for topology-based recoveries (UNREACHABLE → UP).

Version 2 (Multi-parent topology rewrite):
  Completely rewritten to use DeviceLink relationships instead of a
  (now-removed) ParentDeviceId field on Device.
  Added batch heartbeat loading to fix N+1 queries.
  Added isTopologyBasedChange flag to separate heartbeat events from topology events.
  Added UNREACHABLE propagation: if ALL parents of a device are DOWN or UNREACHABLE,
  the device is marked UNREACHABLE (not DOWN).

FINAL ALGORITHM
---------------
1. Load all devices.
2. Load all DeviceLinks with parent devices included.
3. Group DeviceLinks by ChildDeviceId into a dictionary for O(1) lookup.
4. Batch-load latest heartbeat for all devices in a single query.
5. For each device:
   a. Check if device has parents (via DeviceLink dictionary).
   b. If ALL parents are DOWN or UNREACHABLE → newStatus = UNREACHABLE
      (isTopologyBasedChange = true, no heartbeat event logged)
   c. If at least one parent is UP (or device has no parents) →
      check heartbeat timestamp:
        - timeSinceLast > 90s → newStatus = DOWN
          (log HEARTBEAT_TIMEOUT if status is changing)
        - timeSinceLast <= 90s → newStatus = UP
          (log HEARTBEAT_RECOVERED if recovering from DOWN)
   d. Only update device.Status if it changed (avoids unnecessary DB writes).
6. Save all changes in a single batch (context.SaveChangesAsync()).

STATUS DECISION TABLE
---------------------
Parent States                   | Heartbeat     | Resulting Status
--------------------------------|---------------|------------------
At least one parent UP          | < 90s         | UP
At least one parent UP          | > 90s         | DOWN
ALL parents DOWN/UNREACHABLE    | N/A           | UNREACHABLE
No parents (root device)        | < 90s         | UP
No parents (root device)        | > 90s         | DOWN

EVENT LOGGING RULES
-------------------
HEARTBEAT_TIMEOUT:   logged only when a device transitions to DOWN due to
                     heartbeat timeout (not for UNREACHABLE transitions)
HEARTBEAT_RECOVERED: logged only when a device transitions from DOWN to UP
                     due to heartbeat resuming (not for UNREACHABLE → UP)

OPTIMIZATIONS
-------------
- N+1 fix: single grouped query loads all latest heartbeats upfront
  (GetLatestHeartbeatsForAllDevicesAsync added to IHeartbeatRepository)
- Change-only updates: device.Status only written if it differs from newStatus
  (reduces DB writes from N per cycle to only changed devices)

CONFIGURATION
-------------
  Initial delay: 45 seconds (allows scheduler to run first)
  Check interval: 30 seconds
  Failure timeout: 90 seconds (3 missed heartbeats)

8. DEVICE TOPOLOGY FAILURE DETECTION (MULTI-PARENT VIA DEVICELINK)
================================================================================

WHAT
----
Topology-aware failure propagation using the existing DeviceLink table. When a
parent device fails, all child devices that have no other reachable parent are
automatically marked UNREACHABLE instead of DOWN.

WHY
---
Without topology awareness, a single parent failure would cause every downstream
device to appear as independently failed — generating an alarm storm and hiding
the root cause. With topology awareness, operators see one root-cause failure and
all downstream devices correctly marked UNREACHABLE.

ARCHITECTURE DECISION: DeviceLink vs ParentDeviceId
----------------------------------------------------
Initially, adding a ParentDeviceId column to the Device table was considered.
This was abandoned in favour of the existing DeviceLink table because:
- DeviceLink supports multiple parents per device (redundant network paths)
- No schema changes to the Device table required
- Real telecom networks have redundant upstream connections

WHAT WAS REMOVED FROM Device.cs
---------------------------------
- ParentDeviceId (int?) property
- ParentDevice (Device?) navigation property
- Children (ICollection<Device>?) navigation property

WHAT WAS REMOVED FROM AppDbContext.cs
---------------------------------------
- Device-to-Device self-referencing relationship configuration
- HasOne/WithMany/HasForeignKey setup for ParentDeviceId

MULTI-PARENT REDUNDANCY BEHAVIOUR
-----------------------------------
Router A (DOWN) ──┐
                  ├── Device C → remains UP (reachable via Router B)
Router B (UP)   ──┘

Router A (DOWN) ──┐
                  ├── Device C → UNREACHABLE (all paths down)
Router B (DOWN) ──┘

FAILURE PROPAGATION EXAMPLE
-----------------------------
Topology: Core Router (1) → Switch A (2) → Device X (3)

Core Router heartbeat times out (91s):
  Device 1: no parents, heartbeat timeout → DOWN, HEARTBEAT_TIMEOUT logged
  Device 2: parent = Device 1 (DOWN), all parents down → UNREACHABLE, no event
  Device 3: parent = Device 2 (UNREACHABLE), all parents down → UNREACHABLE, no event

Result: operators see 1 root-cause alert, not 3 independent failures.

RECOVERY PROPAGATION EXAMPLE
------------------------------
Core Router recovers (heartbeat resumes):
  Device 1: heartbeat < 90s, was DOWN → UP, HEARTBEAT_RECOVERED logged
  Device 2: parent = Device 1 (UP), heartbeat < 90s, was UNREACHABLE → UP, HEARTBEAT_RECOVERED logged
  Device 3: parent = Device 2 (UP), heartbeat < 90s, was UNREACHABLE → UP, HEARTBEAT_RECOVERED logged

10. DEVICESTATUS ENUM MODIFICATIONS
================================================================================

WHAT
----
IMPACTED and UNKNOWN values removed from the DeviceStatus enum as they were unused.

WHY
---
The topology failure detection logic uses UNREACHABLE to distinguish between
a device that failed on its own (DOWN) versus one that is inaccessible because
its parent failed (UNREACHABLE). IMPACTED and UNKNOWN were not being set in the code.

FILE
----
- INMS.Domain/Enums/DeviceStatus.cs [MODIFIED]

VALUES REMOVED
------------
IMPACTED — unused status
UNKNOWN  — unused status

FULL ENUM
---------
public enum DeviceStatus
{
    UP,           // Device responding normally (heartbeat within 90s)
    DOWN,         // Device itself failed (heartbeat timeout)
    UNREACHABLE   // All parents DOWN/UNREACHABLE — topology-based
}
